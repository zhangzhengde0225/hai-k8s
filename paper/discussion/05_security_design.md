# 安全设计讨论

Author: 白小心 & 朵拉
Date: 2026-03-28

## 背景：CNCERT OpenClaw 安全风险提示

2026年3月，国家互联网应急中心（CNCERT）联合中国网络空间安全协会发布了 OpenClaw 安全风险提示（央视网 2026-03-10），指出了四大安全风险：

1. **提示词注入**：网页中隐藏恶意指令诱导 OpenClaw 泄露密钥
2. **误操作风险**：OpenClaw 误解指令删除重要数据
3. **Skills 插件投毒**：恶意 Skills 可窃取密钥、部署木马
4. **安全漏洞**：OpenClaw 已公开多个高中危漏洞

**CERT 核心建议**：使用 Docker/容器隔离、最小权限、Skills 签名验证、环境隔离。

## HAI-K8S 如何回应 CERT 安全建议

## 论文中安全部分的内容建议

### 5.1 安全威胁模型（Threat Model）

基于 CERT 识别的威胁：
- 攻击面：提示词注入、恶意 Skills、漏洞利用、权限滥用
- 影响范围：密钥泄露、数据删除、系统被控

### 5.2 安全设计原则

1. **纵深防御**：多层隔离（K8s namespace → Pod → Container → User namespace）
2. **最小权限**：容器只申请必要权限，Agent 只访问必要目录
3. **零信任认证**：两层认证（Admin API Key + User JWT）
4. **可审计**：所有操作可追溯（K8s audit logs + skill exec logs）

### 5.3 安全实验设计

**实验4：容器隔离有效性**

研究问题：容器化是否有效阻止恶意 Skills 的横向扩散？

对比方案：
- A) HAI-K8S 容器隔离（Pod 独立）
- B) 非隔离环境（直接运行在宿主机）

攻击模拟：
1. 恶意 Skill 尝试读取其他用户的数据
2. 恶意 Skill 尝试获取宿主机的 SSH 密钥
3. 恶意 Skill 尝试修改系统文件

测量指标：
- 攻击成功率
- 横向扩散范围
- 系统完整性（是否被完全控制）

**预期结果**：
- 容器隔离下攻击者被困在 Pod 内，无法横向扩散
- 非隔离环境下攻击者可以获得宿主机的完全控制

### 5.4 安全最佳实践

结合 CERT 建议 + HAI-K8S 实现，给出安全部署最佳实践：
1. 强制容器隔离（不过度授权）
2. Skills 来源白名单
3. Gateway 不暴露公网
4. 操作审计日志
5. 定期更新 OpenClaw 版本
